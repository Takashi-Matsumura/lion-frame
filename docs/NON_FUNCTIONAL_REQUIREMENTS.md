# 非機能要件定義書

LionFrame — 組織管理システムの非機能要件を定義する。

> **対象環境:** 社内LAN上のサーバ（Docker Compose構成）
> **利用端末:** 社内PC（ブラウザ）
> **想定規模:** 従業員50〜300名程度の組織

---

## 1. 可用性（止まらないこと）

| 項目 | 要件 |
|------|------|
| 稼働時間帯 | 平日 9:00〜18:00 |
| 計画停止 | 業務時間外に実施。1営業日前にアナウンスメント通知で周知 |
| 障害復旧目標 | 翌営業日の午前中までに復旧（金曜障害 → 月曜午前を含む） |
| メンテナンス中の表示 | アナウンスメント通知で利用者に状況を伝える（専用画面は不要） |
| モジュール稼働監視 | 管理者がモジュールの稼働状態と影響範囲を把握できること |

### 機能として必要なもの

| 状態 | 機能 |
|------|------|
| 実装済 | アナウンスメント通知（計画停止の事前周知・障害告知） |
| 実装済 | モジュールヘルスダッシュボード（稼働状態の一覧・影響分析） |
| 実装済 | 障害通知テンプレート（ヘルスチェック結果からアナウンス作成） |

---

## 2. バックアップ・リカバリ（データを失わないこと）

| 項目 | 要件 |
|------|------|
| バックアップ頻度 | 1日1回（深夜、業務時間外） |
| バックアップ方式 | `pg_dump` によるフルバックアップ（Docker環境用スクリプト整備） |
| 保持期間 | 日次バックアップ：30日分 |
| 年度末スナップショット | 組織改編前にスナップショットを取得し、年度単位で保持 |
| リカバリ目標時間（RTO） | 2時間以内 |
| リカバリ目標地点（RPO） | 前日深夜のバックアップ時点 |
| 監査ログの扱い | 監査ログは欠損を許容しない。バックアップ間のログもDB上で保全する |
| バックアップ検証 | 四半期に1回、リストアテストを実施 |

### 機能として必要なもの

| 状態 | 機能 |
|------|------|
| 要検討 | バックアップ状態の管理画面（最終バックアップ日時、成否の表示） |
| 要検討 | `pg_dump` / リストア用スクリプトの整備 |

---

## 3. パフォーマンス（快適に使えること）

| 項目 | 要件 |
|------|------|
| 初回画面表示（Cold） | 3秒以内（ログイン後の初回ページ読み込み） |
| ページ間遷移（Warm） | 1秒以内 |
| 一覧検索 | 2秒以内 |
| CSVインポート | 500件を30秒以内 |
| CSVインポート（大量） | 1万件超の場合はバックグラウンド処理に切り替え、完了を通知 |
| AIチャット応答 | 外部API依存のため別基準。初回応答開始まで10秒以内を目安 |
| 同時利用者数 | 50名程度（ピーク：月曜朝の一斉ログイン等） |

### 機能として必要なもの

| 状態 | 機能 |
|------|------|
| 実装済 | UsageLog / UsageStat（利用状況の記録） |
| 要検討 | スロークエリの検出・記録 |
| 要検討 | 大量インポート時のバックグラウンド処理 + 通知 |

---

## 4. セキュリティ（守ること）

| 項目 | 要件 |
|------|------|
| 認証方式 | ID／パスワード認証。OAuth（Google等）は管理画面で有効化可能 |
| 2要素認証（2FA） | 将来対応（TOTP方式を想定） |
| パスワードポリシー | 最低8文字、英数字混合。有効期限・再利用禁止は運用に応じて設定 |
| アカウントロック | ログイン連続5回失敗で30分ロック |
| セッション管理 | 60分無操作でセッションタイムアウト（管理画面で変更可能） |
| 権限分離 | ロール別アクセス制御（GUEST / USER / MANAGER / EXECUTIVE / ADMIN） |
| 通信暗号化 | HTTPS必須（リバースプロキシで終端） |
| 監査ログ | ログイン、ログアウト、重要操作（データ変更・削除・エクスポート等）を記録 |
| 個人情報保護 | 従業員データの閲覧はロールに応じて制限。エクスポートは管理者のみ |

### 機能として必要なもの

| 状態 | 機能 |
|------|------|
| 実装済 | ロール別アクセス制御 |
| 実装済 | 監査ログ（AuditLog） |
| 要検討 | セッションタイムアウト設定（管理画面） |
| 要検討 | パスワードポリシー設定 |
| 要検討 | ログイン失敗によるアカウントロック |
| 要検討 | 2FA（TOTP）実装 |

---

## 5. 保守性（直しやすいこと）

| 項目 | 要件 |
|------|------|
| ログレベル | ERROR / WARN / INFO の3段階 |
| ログ形式 | 構造化ログ（JSON形式）で出力し、検索・分析を可能にする |
| ログ保持期間 | 90日 |
| 環境再構築 | Docker Compose + 手順書により2時間以内に再構築可能 |
| アップデート頻度 | 月1回程度、業務時間外に実施 |
| テスト方針 | 主要APIの自動テスト（契約テスト）を維持し、リグレッションを防止 |
| ドキュメント | 運用手順書、障害対応手順書、環境構築手順書を整備 |
| ヘルスチェック | `/api/health` エンドポイントでアプリケーション状態を確認可能に |

### 機能として必要なもの

| 状態 | 機能 |
|------|------|
| 実装済 | システム診断機能（DB、認証、通知等の6項目チェック） |
| 要検討 | 構造化ログ出力（JSON形式） |
| 要検討 | ヘルスチェックエンドポイント（`/api/health`） |
| 要検討 | アプリケーションログの閲覧画面（管理者向け） |

---

## 6. 信頼性（壊れにくいこと）

| 項目 | 要件 |
|------|------|
| データ整合性 | トランザクション処理による一貫性の保証 |
| 排他制御 | 同一データの同時編集時は楽観的ロック（更新時のバージョンチェック）で競合を検出 |
| データ削除ポリシー | 重要データは論理削除（削除フラグ）。物理削除は管理者操作 or バッチ処理 |
| プロセス監視 | Docker `restart: always` + ヘルスチェック付き再起動 |
| ディスク監視 | 残容量が閾値を下回った場合にアラート |
| DB接続監視 | PostgreSQLのコネクション数を監視し、上限接近時にアラート |

### 機能として必要なもの

| 状態 | 機能 |
|------|------|
| 実装済 | Docker restart policy |
| 要検討 | 楽観的ロック（更新時のバージョンチェック） |
| 要検討 | DB接続プールの監視（診断機能に追加） |
| 要検討 | ディスク容量の監視（診断機能に追加） |

---

## 7. 移行性（将来に備えること）

| 項目 | 要件 |
|------|------|
| データエクスポート | 各一覧画面からCSV / Excelで出力可能 |
| 環境移行 | Docker Compose構成により、別サーバへの移行が可能 |
| DBマイグレーション | Prisma Migrateによるスキーマ変更の安全な適用手順を整備 |
| スケールアップ判断 | 1年後の判断材料として以下を継続的に収集する |
| 外部連携 | 将来の他システムとのデータ連携を考慮し、APIを整備する |

### スケールアップ判断の収集指標

| 指標 | 収集方法 |
|------|---------|
| 月間アクティブユーザ数の推移 | UsageStat |
| DB容量の増加率 | 診断機能 or 運用スクリプト |
| レスポンスタイムの推移 | UsageLog / アクセスログ |
| 機能別利用頻度 | UsageLog |

### 機能として必要なもの

| 状態 | 機能 |
|------|------|
| 実装済 | データエクスポート（CSV / Excel） |
| 実装済 | UsageStat（利用状況の収集） |
| 要検討 | DB容量モニタリング（診断機能に追加） |

---

## 8. 互換性（使える環境）

| 項目 | 要件 |
|------|------|
| 対応ブラウザ | Google Chrome 最新版、Microsoft Edge 最新版 |
| 対応OS | Windows 10以降（社内PC標準） |
| モバイル対応 | Webアプリはデスクトップ専用。モバイルは別途React Nativeアプリで対応予定 |
| 画面解像度 | 1920x1080（Full HD）を基準。1366x768以上で操作可能 |

---

## 9. キャパシティ（データ量の目安）

| 項目 | 上限目安 |
|------|---------|
| 従業員マスタ | 1,000名 |
| 監査ログ | 100万件（超過分は年次アーカイブ） |
| 通知データ | 既読後90日経過で自動削除 |
| 添付ファイル / ドキュメント | 合計10GB |
| DBサイズ全体 | 50GB |

---

## 10. 運用体制

| 項目 | 要件 |
|------|------|
| 管理者人数 | 1〜2名（ADMINロール） |
| 障害時の連絡フロー | 管理者が検知 → 原因調査 → 必要に応じてアナウンスメント通知 |
| 問い合わせ対応 | 社内ヘルプデスク or 管理者が対応 |
| 定期作業 | バックアップ確認（日次）、ログ確認（週次）、アップデート適用（月次） |

| パスワードポリシー設定 | セキュリティ |
| `pg_dump` / リストア用スクリプト | バックアップ・リカバリ |
| ヘルスチェックエンドポイント | 保守性・信頼性 |

### 優先度：中（運用品質の向上）

| 機能 | 関連する非機能要件 |
|------|-------------------|
| ~~メンテナンスモード画面~~ | ~~可用性~~（モジュールヘルスチェック + アナウンスで代替済） |
