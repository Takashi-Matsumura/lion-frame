# ハンズオン教材作成ガイド

LionFrameのエディタ（マークダウン）で作成し、ハンズオンモジュールで受講者に配信する教材の作成ルール。

## 教材の仕組み

```
マークダウン教材 → パーサーが自動解析 → インタラクティブ教材として配信
                    ↓
            セクション / ステップ / コードブロック に分割
            コードブロックにOK/Errorボタンが自動付与
            コラムは折りたたみ表示
```

教材は通常のマークダウンだが、**パーサーが認識する特定の記法**に従う必要がある。
記法に従えば、受講者画面で自動的にインタラクティブ要素（進捗ボタン、折りたたみコラム等）が付与される。

## 必須記法ルール

### 1. セクション分割: `---`（水平線）

水平線（`---`）で教材をセクションに分割する。各セクションは独立したブロックとして表示される。

```markdown
# タイトル

導入テキスト...

---

## 1. 最初のセクション

### 1-1. ステップ1

手順の説明...

---

## 2. 次のセクション

### 2-1. ステップ1

手順の説明...
```

**注意:**
- `---` は前後に空行を入れること（空行がないとマークダウンのヘッダー装飾と誤認される場合がある）
- 最初のセクション（タイトル・導入部分）は `---` の前に書く

### 2. ステップ識別: `### N-N. タイトル`

各ステップは `### セクション番号-ステップ番号. タイトル` の形式で書く。

```markdown
### 1-1. GitHubにサインアップする

1. ブラウザで https://github.com にアクセスします
2. 右上の「Sign up」をクリックします

### 1-2. SSH鍵を生成する

ターミナルで以下のコマンドを実行します：
```

**パターン:** `### N-N. テキスト`（正規表現: `/^###\s+(\d+-\d+)\.\s+(.+)$/`）

| 形式 | 認識 |
|------|------|
| `### 1-1. タイトル` | OK |
| `### 2-3. タイトル` | OK |
| `## 1-1. タイトル` | NG（`##` はステップとして認識されない） |
| `### 1. タイトル` | NG（`N-N.` の形式が必要） |
| `### 1-1 タイトル` | NG（ピリオドが必要） |

### 3. コードブロック: 自動インデックス付与

フェンスコードブロック（` ``` `）はパーサーが自動検出し、グローバルインデックスを付与する。
受講者画面では各コードブロックの下に **OK / Error ボタン** が自動表示される。

````markdown
### 1-1. Gitの初期設定

ユーザ名を設定します：

```bash
git config --global user.name "あなたの名前"
```

メールアドレスを設定します：

```bash
git config --global user.email "your-email@example.com"
```
````

上記の場合、2つのコードブロックにグローバルインデックス 0, 1 が付与される。
講師画面ではこのインデックスで受講者の進捗を追跡する。

**重要:**
- コードブロックの言語指定（`bash`, `python` 等）を必ず書く（シンタックスハイライトに影響）
- イントロ部分（最初のステップの前）のコードブロックはインデックス付与されない
- 1つのステップに複数のコードブロックを含めてよい

### 4. コラム: `### コラム：タイトル`

補足情報は「コラム」として折りたたみ表示できる。セクション全体をコラムとして扱う。

```markdown
---

### コラム：Gitのブランチ戦略について

ブランチとは、開発の流れを分岐させる機能です。
主なブランチ戦略には以下があります：

- **Git Flow**: 大規模プロジェクト向け
- **GitHub Flow**: シンプルで小規模チーム向け

---
```

**パターン:** `### コラム：タイトル` または `### コラム:タイトル`（全角・半角コロン両対応）

**注意:**
- コラムは独立したセクション（前後を `---` で区切る）として書く
- コラムのコードブロックにはOK/Errorボタンは付かない
- 受講者画面ではアコーディオン（クリックで展開）として表示される

## 教材の全体構成テンプレート

```markdown
# 教材タイトル：サブタイトル

## 今日のゴール

ゴールの説明文。この教材で何ができるようになるかを簡潔に書く。

---

## 1. 最初の大きなテーマ（目安時間）

### 1-1. 最初のステップ

説明テキスト。

```bash
実行するコマンド
```

### 1-2. 次のステップ

説明テキスト。

```bash
実行するコマンド
```

---

### コラム：関連する補足知識

補足説明...

---

## 2. 次の大きなテーマ（目安時間）

### 2-1. ステップ

...

---

## まとめ

- 学んだこと1
- 学んだこと2
- 学んだこと3

### 次回予告

次回は「○○」について学びます。
```

## 教材作成の注意事項

### 構成設計

| ルール | 理由 |
|--------|------|
| 1セクションは30〜50分を目安にする | 受講者の集中力が持続する単位 |
| 1ステップは1つの操作に集中する | 「何をすべきか」が明確になる |
| コードブロックは1コマンド＝1ブロックにする | OK/Errorの粒度が細かくなり、講師が詰まりポイントを特定しやすい |
| セクション冒頭にゴールを書く | 受講者が「なぜこの操作をするか」を理解できる |

### 文章スタイル

| ルール | 例 |
|--------|-----|
| 操作手順は番号付きリストで書く | `1. 画面右上の「設定」をクリックします` |
| コマンドの前に「何をするか」を1行書く | `ユーザ名を設定します：` → コードブロック |
| 結果の確認方法を必ず書く | `以下のように表示されれば成功です：` |
| 専門用語は初出時に説明する | `リポジトリ（プロジェクトの保存場所）を作成します` |
| 「です・ます」調で統一する | 丁寧語で統一 |

### コードブロックの書き方

```markdown
<!-- ✅ 良い例: 1コマンド1ブロック、説明付き -->

リポジトリを初期化します：

```bash
git init
```

最初のコミットを作成します：

```bash
git add .
git commit -m "最初のコミット"
```

<!-- ❌ 悪い例: 複数の無関係なコマンドを1ブロックに詰め込む -->

```bash
git init
git add .
git commit -m "最初のコミット"
git remote add origin https://...
git push -u origin main
```
```

**例外:** `git add` + `git commit` のように、セットで実行するコマンドは1ブロックにまとめてよい。

### 期待結果の示し方

コマンド実行後の期待結果は、コードブロックではなく引用ブロックやテキストで示す。
（コードブロックにするとOK/Errorボタンが付いてしまう）

```markdown
```bash
git status
```

> 以下のような出力が表示されます：
> ```
> On branch main
> nothing to commit, working tree clean
> ```
```

### エディタでの作成手順

1. **エディタメニュー** → 「新規作成」→「マークダウン」を選択
2. タイトルを設定（例: `Git ハンズオン Day 1：書いて、記録して、公開する`）
3. **ライブプレビュー**モードで執筆（リアルタイムで見た目を確認）
4. 完成したら**ステータスを「公開」**に変更
5. ハンズオンセッション作成時にこのドキュメントを選択

**ステータスが「公開」でないとハンズオンセッションで選択できない。**

### PDF出力との両立

教材は受講者が手元で見返すためにPDF出力されることがある。

- `---`（セクション区切り）はPDF上で水平線として表示される
- コラム（`### コラム：`）はPDFでは折りたたまれず通常表示される
- 画像を使う場合は、エディタの「メディア」機能でアップロードする

## チェックリスト

教材作成後、以下を確認する：

- [ ] タイトルに「Day N」や日付を含め、シリーズの位置がわかるようにしているか
- [ ] 冒頭に「今日のゴール」を書いているか
- [ ] `---` でセクションが正しく区切られているか
- [ ] ステップが `### N-N. タイトル` の形式になっているか
- [ ] 各コードブロックに言語指定があるか（`bash`, `python` 等）
- [ ] 1コマンド＝1コードブロックの粒度になっているか
- [ ] 各コマンドの前に「何をするか」の説明があるか
- [ ] 期待結果の確認方法が書かれているか
- [ ] 補足情報はコラム（`### コラム：`）に分離しているか
- [ ] ドキュメントのステータスが「公開」になっているか
