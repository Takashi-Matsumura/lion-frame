テスト計画の作成・テスト状況の確認を行います。

## サブコマンド判定

引数（$ARGUMENTS）を確認し、以下のルールで動作を分岐する:

- `help` → **ヘルプモード** に進む
- `show` → **テスト一覧モード** に進む
- `show [ファイル名やキーワード]` → **テスト一覧モード（フィルタ付き）** に進む
- それ以外（ファイルパス、機能名、モジュール名など） → **テスト計画モード** に進む
- 引数なし → ユーザに「何のテストを作成しますか？」と聞く

---

## ヘルプモード（help）

以下をそのまま表示する:

```
## /test-plan コマンドの使い方

テストの計画・作成・確認を行うコマンドです。

### コマンド一覧

| コマンド | 説明 |
|---------|------|
| /test-plan help | このヘルプを表示 |
| /test-plan show | 全テストの一覧と合否を表示 |
| /test-plan show [キーワード] | キーワードでフィルタしてテスト一覧を表示 |
| /test-plan [対象] | 指定した対象のテスト計画を作成 |

### 対象の指定方法

- ファイルパス: `/test-plan lib/api/auth-guard.ts`
- 機能名: `/test-plan ロール階層チェック`
- モジュール名: `/test-plan organization`

### テスト計画の流れ

1. 対象のコードを分析
2. テストケースを日本語の表で提案（コードは書かない）
3. あなたに確認を求める（追加・削除・変更のリクエスト可能）
4. 承認後にテストコードを実装・実行
5. 結果を報告

計画段階であなたが判断するのは:
- このテストは本当に必要か？
- 抜けているケースはないか？
- 重要度（必須/推奨/任意）は妥当か？
```

ヘルプを表示したら終了。他のモードには進まない。

## テスト一覧モード（show）

`apps/web/__tests__/` 配下の全テストファイルを調査し、以下のフォーマットで報告する。

### 手順

1. `apps/web/__tests__/` 配下の全 `.test.ts` / `.test.tsx` ファイルを探す
2. 各ファイルを読んで、`describe` と `it` / `test` の構造を抽出する
3. テストを実行して現在の合否状態を取得する（`npx jest --no-coverage` を使用）
4. 以下のフォーマットで表示する

### 出力フォーマット

```
## テスト一覧

### 全体サマリ
- テストファイル: X 個
- テストケース: 合計 X 件（合格 X / 失敗 X）

### ファイル別

#### [テストファイルのパス]
対象: [テスト対象のソースファイル]

| # | テスト内容 | 結果 |
|---|-----------|------|
| 1 | ...       | PASS |
| 2 | ...       | PASS |

（ファイルごとに繰り返す）

### テストがない主要ファイル
[テストが存在しないが、テストがあるべき重要なファイルを最大5つ挙げる]
```

フィルタ付き（`show auth-guard` など）の場合は、キーワードに一致するテストファイルのみ表示する。

## テスト計画モード

ユーザが指定した対象（ファイル、機能、モジュール名など）に対して、テストの計画を立てる。
**いきなりテストコードを書かないでください。** 必ず計画フェーズを経てから実装に進みます。

## 手順

### Phase 1: 対象コード分析

1. 指定された対象のソースコードを読む
2. そのコードが何をしているかを簡潔に説明する
3. 既存のテストがあるか確認する

### Phase 2: テスト計画の提案

以下のフォーマットで **日本語の表** として提案する。コードは書かない。

```
## テスト計画: [対象名]

### 対象コードの概要
[1-2行で何をするコードか説明]

### 既存テストの状況
[既存テストの有無と、カバーされている範囲]

### 提案するテストケース

| # | カテゴリ | テスト内容 | 期待結果 | 重要度 | 理由 |
|---|---------|-----------|---------|--------|------|
| 1 | 正常系   | ...       | ...     | 必須   | ... |
| 2 | 異常系   | ...       | ...     | 必須   | ... |
| 3 | 境界値   | ...       | ...     | 推奨   | ... |

重要度の基準:
- 必須: バグが出たらセキュリティやデータに影響する
- 推奨: バグが出たら機能が壊れる
- 任意: あると安心だが、なくても大きな問題にはならない

### テストしない項目（と理由）
[あえてテストしないものを明示し、その理由を説明]
```

### Phase 2 で止まる

計画を提示したら **必ずユーザに確認を求める**。以下のように聞く:

「この計画について確認してください。
- 追加したいテストケースはありますか？
- 不要だと思うテストケースはありますか？
- 重要度を変更したいものはありますか？
- その他、気になる点はありますか？

確認が取れたら実装に進みます。」

### Phase 3: テスト実装（承認後のみ）

ユーザが計画を承認したら:

1. テストスキル（`.claude/skills/testing/SKILL.md`）に従ってテストコードを書く
2. 実装したテストを実行する
3. 結果を報告する

### Phase 4: 結果報告

```
## テスト結果

| # | テスト内容 | 結果 | 備考 |
|---|-----------|------|------|
| 1 | ...       | PASS |      |
| 2 | ...       | PASS |      |

全 X テスト中 X テスト合格
```

## 注意事項

- ユーザはテスト未経験者に近い。専門用語を使う場合は簡単な説明を添える
- テストケースの「テスト内容」は技術的すぎず、日本語で何を確認するか伝わるように書く
- 「モック」「スタブ」などの用語は、必要な場合のみ簡単に説明する
- 計画段階ではコードを見せない（実装段階で初めてコードを書く）
