テスト駆動開発（TDD）のワークフローを強制します。

## 概要

RED → GREEN → REFACTOR のサイクルに従い、**テストを先に書いてから実装**する。
既存の `/test-plan` とは役割が異なる:
- `/test-plan` = テスト計画の立案（何をテストするか決める）
- `/tdd` = TDDサイクルでの実装（テストファーストで書く）

## ワークフロー

### Step 1: 対象の理解
- 引数（$ARGUMENTS）で指定された対象のコードを読む
- 何を実装/修正するのかを明確にする

### Step 2: インターフェース定義
- 入出力の型・インターフェースを先に定義する
- 実装コードはまだ書かない

### Step 3: RED（失敗するテストを書く）
- 期待する振る舞いを表現するテストを書く
- `.claude/skills/testing/SKILL.md` のパターンに従う
- テストを実行し、**失敗することを確認する**

```bash
cd apps/web && npx jest --no-coverage [テストファイル]
```

### Step 4: GREEN（最小限の実装）
- テストを通す**最小限のコード**だけを書く
- 余計な機能や最適化はこの段階では不要

```bash
cd apps/web && npx jest --no-coverage [テストファイル]
```

### Step 5: REFACTOR（リファクタリング）
- テストが通った状態でコードを整理する
- 重複の除去、命名の改善、構造の最適化
- テストが緑のまま維持されていることを確認

### Step 6: 繰り返し
- 次のテストケースに進み、Step 3-5 を繰り返す
- すべてのケースが完了するまで続ける

### Step 7: 最終確認
- 全テストを実行して合格を確認

```bash
cd apps/web && npx jest --no-coverage [テストファイル]
```

## テストで必ずカバーすべきケース

| カテゴリ | 内容 |
|---------|------|
| 正常系 | 期待通りの入力での動作 |
| 異常系 | 不正な入力、null/undefined |
| 境界値 | 空配列、空文字列、0、最大値 |
| エラーパス | DB障害、認証失敗、404 |

## LionFrame固有のルール

- テストファイルは `apps/web/__tests__/` 配下に配置
- モック方法は `.claude/skills/testing/SKILL.md` に従う
- Prisma のモックは `@/lib/__mocks__/prisma` を使用
- 認証のモックは `@/lib/__mocks__/auth` を使用
- 翻訳テストは不要（i18nはフレーム基盤で保証）

## やってはいけないこと

- テストより先に実装コードを書く
- 一度に大量のテストケースを書く（1つずつ RED→GREEN→REFACTOR）
- 実装の内部詳細をテストする（振る舞いをテストする）
- テスト間で状態を共有する
